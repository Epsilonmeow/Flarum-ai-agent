# Flarum AI Agent 产品路线图

> 本文档用于沉淀项目核心方向、产品边界、阶段规划、验收标准与技术任务拆解。当前规划基于现有 Flarum AI Agent、V2 工业级升级、Streamlit 管理后台、Flarum Webhook、世界书 V3、三层记忆、好感度系统、沙盒测试，以及可选外部助手探测原型。

---

## 1. 产品愿景

Flarum AI Agent 的长期愿景不是做一个简单的自动回复机器人，而是构建一个面向社区场景的 **人格化 AI 陪伴与信息助手平台**。

它应当具备以下核心能力：

1. **有温度的长期陪伴**：在论坛树洞、社群聊天等场景中，以稳定人格持续回应用户表达。
2. **可控的长期记忆**：通过三层记忆、向量检索、用户画像与好感度系统，让 AI 能逐步理解用户，但不无边界地记住所有内容。
3. **可运营的知识与人格系统**：通过世界书、Prompt、规则、沙盒测试与后台管理，让非开发者也能调整 AI 的表达风格、知识边界与触发逻辑。
4. **安全优先的自动化链路**：在真实论坛或群聊环境中，始终坚持沙盒先行、灰度接入、冷却限制、自循环防护、敏感内容升级与日志审计。
5. **多渠道校园信息中枢**：逐步从 Flarum 论坛扩展到微信群、文件、OCR、日报/周报等信息流，形成“陪伴 + 总结 + 检索 + 提醒”的校园 AI 助手。

一句话定位：

> 用可测试、可运营、可控的 AI Agent，温柔地承接校园社群里的表达、记忆与信息流。

## 2. 核心需求判断

### 2.1 显性需求

- 在 Flarum 论坛中接收 Webhook 事件并自动生成回复。
- 使用世界书为 AI 注入结构化知识、人格设定和场景规则。
- 使用记忆系统保存有价值的用户互动，支持长期上下文。
- 使用好感度系统表达 AI 与用户关系的变化。
- 使用 Streamlit 后台管理配置、世界书、Provider、记忆、好感度与测试结果。
- 使用沙盒测试在不调用真实 LLM、不发帖、不污染正式数据的前提下验证主流程。

### 2.2 隐性需求

- **可信赖**：真实论坛环境不能不可控试错，必须先沙盒、后灰度、再正式。
- **可解释**：运营者需要知道一次回复为什么这样生成，包括命中的世界书、注入的记忆、使用的 Provider、是否写入记忆、好感度如何变化。
- **可维护**：模型、Prompt、世界书、记忆、发帖逻辑、测试用例应当解耦，避免单文件膨胀和隐式副作用。
- **可扩展**：后续要能扩展到微信、文件、OCR、日报周报，而不是只服务单一 Flarum Webhook。
- **有边界**：对于心理危机、隐私、恶意刷屏、自循环、低价值内容，系统必须有明确策略。

### 2.3 产品核心方向

当前项目的核心方向可以拆成三层：

| 层级 | 核心问题 | 当前能力 | 下一步重点 |
|------|----------|----------|------------|
| AI 角色层 | 树洞喵是谁，如何稳定表达 | System Prompt、世界书、输出正则 | 人格评估集、Prompt 预览、回复质量标注 |
| 记忆关系层 | 它如何长期理解用户 | 三层记忆、好感度、Embedding | 记忆写入解释、用户画像审核、隐私删除闭环 |
| 运营安全层 | 如何安全上线和持续迭代 | 沙盒、MockProvider、冷却、后台 | pytest、灰度发布、人工审核、日志审计 |

## 3. 用户画像

### 3.1 终端用户：论坛/社群成员

典型用户：

- 在校园论坛或树洞中表达压力、困惑、情绪或日常想法的学生。
- 希望得到及时、温柔、低压力回应的人。
- 不一定希望被“解决问题”，但希望被倾听、被理解、被适度引导。

核心诉求：

- 回复要自然、温和，不像生硬客服。
- AI 能记得长期互动中的重要信息，但不要过度窥探隐私。
- 遇到危机表达时，能提供更谨慎、更可靠的支持路径。
- 不希望机器人刷屏或频繁打扰。

成功体验：

- “它记得我之前提过的事情。”
- “它回复得不冒犯，也不会乱下判断。”
- “它像一个稳定在线的温柔角色。”

### 3.2 运营用户：社群/论坛管理员

典型用户：

- 论坛管理员、社群运营者、项目维护者。
- 负责配置 Bot、维护世界书、观察回复质量、处理异常情况。

核心诉求：

- 能看懂系统当前是否健康。
- 能知道某条回复为什么产生。
- 能安全调整世界书、Prompt、Provider 和冷却策略。
- 能在真实环境上线前完成充分测试。
- 能处理敏感内容、误回复、自循环和用户删除请求。

成功体验：

- “我能在后台定位问题，而不是只能看日志猜。”
- “每次上线前我可以先跑沙盒。”
- “如果 AI 不确定，我可以人工审核。”

### 3.3 开发用户：项目开发者

典型用户：

- 维护核心引擎、Provider、记忆系统、Webhook、后台和微信助手的开发者。

核心诉求：

- 模块边界清晰，便于扩展。
- 有稳定测试集，避免改一处坏三处。
- 数据目录、沙盒目录、正式目录明确隔离。
- 关键链路可复现、可调试、可回归。

成功体验：

- “我能快速验证一个 Prompt 或世界书改动。”
- “我能独立测试 Provider、记忆、好感度和 Webhook。”
- “我能把 Flarum 能力迁移到微信等新渠道。”

## 4. 功能分层

### 4.1 L0 基础设施层

目标：保证项目能稳定启动、部署、配置和测试。

现有能力：

- FastAPI Webhook 服务。
- Dockerfile 与 docker-compose。
- `.env` 配置加载。
- Streamlit 管理后台。
- MockProvider 离线测试。
- 沙盒数据目录隔离。

下一步：

- 标准化 pytest 测试入口。
- 增加健康检查细分项。
- 增加配置合法性检查和 `.env` 错误提示。
- 增加日志结构化输出。

### 4.2 L1 论坛回复层

目标：稳定完成 Flarum 论坛监听、识别、回复与冷却控制。

现有能力：

- Webhook 异步处理。
- Flarum API 发帖。
- 自循环防护。
- 回复冷却机制。
- 记忆价值过滤。

下一步：

- 增加白名单/黑名单/板块范围控制。
- 增加人工审核模式。
- 增加回复失败重试与死信记录。
- 增加 Webhook payload 回放测试。

### 4.3 L2 AI 生成层

目标：让 AI 回复稳定、有边界、有角色一致性。

现有能力：

- ProviderManager 多模型故障转移。
- OpenAI-compatible Provider。
- System Prompt。
- 世界书匹配。
- 输出正则处理。

下一步：

- 增加 Prompt 预览与上下文解释。
- 增加回复质量评估集。
- 增加敏感场景专用模板。
- 增加 Provider 调用成本与失败率统计。

### 4.4 L3 记忆与关系层

目标：让树洞喵拥有长期陪伴感，同时保持隐私与可控性。

现有能力：

- L1 摘要层、L2 块层、L3 历史层。
- 增量总结策略。
- 好感度等级与奖励规则。
- JSON 原子写入、并发锁、损坏备份。
- ChromaDB 旧版向量记忆能力。

下一步：

- 增加记忆写入原因解释。
- 增加用户画像查看、编辑、删除与导出。
- 增加“低价值内容不写入”的可视化说明。
- 明确 ChromaDB 与 V2 JSON 记忆的职责边界。

### 4.5 L4 运营后台层

目标：让非开发者可以运营、测试、诊断和调优 AI。

现有能力：

- 系统状态页。
- Provider 管理。
- 沙盒测试页。
- 世界书 V3 编辑。
- 三层记忆查看。
- 好感度系统查看。
- 系统配置与冷却监控。

下一步：

- 增加单次回复链路追踪。
- 增加沙盒对话回放。
- 增加世界书命中解释。
- 增加测试用例编辑器。
- 增加灰度发布控制台。

### 4.6 L5 多渠道信息助手层

目标：从论坛陪伴扩展为校园多源信息助手。

现有能力：

- `wechat_group_assistant` 只读探测原型。
- 微信窗口检测、群名识别、可见文本探测、文件目录定位。

下一步：

- `collector`：低频采集群文字消息，去重写入 JSONL。
- `attachment_watcher`：监听微信文件目录并归档 PDF/docx/xlsx。
- `document_parser`：解析附件正文。
- `summarizer`：分类、打标签、生成日报/周报。
- `safe_ocr`：在安全模式下进行截图与 OCR。

## 5. 里程碑规划

### Milestone 0：当前基线状态

状态：已基本完成。

已具备：

- Flarum Webhook 主链路。
- V2 LLMEngine。
- Provider 故障转移。
- 三层记忆系统。
- 好感度系统。
- 世界书 V3。
- Streamlit 后台。
- MockProvider 沙盒测试。
- JSON 存储稳定性保护。

主要缺口：

- 完整 pytest 测试套件不足。
- 回复链路可解释性不足。
- 灰度上线和人工审核能力不足。
- 微信助手仍处于只读探测原型。

### Milestone 1：稳定化与自动测试

周期建议：1–2 周。

目标：让核心链路具备可回归、可验证、可持续迭代的基础。

交付物：

- pytest 测试套件。
- Webhook payload 回放测试。
- 世界书扫描测试。
- Provider failover 测试。
- 三层记忆与好感度回归测试。
- 沙盒测试报告结构固定化。

验收标准：

- 本地运行 `pytest` 能覆盖核心模块。
- `python sandbox_test.py` 仍保持通过。
- MockProvider 下不调用真实 LLM、不发帖、不污染正式数据。
- 修改世界书、Provider、记忆逻辑后能快速发现回归问题。

### Milestone 2：后台可解释性增强

周期建议：1–2 周。

目标：让运营者能看懂一次回复的生成过程。

交付物：

- 单次回复链路追踪视图。
- Prompt 上下文预览。
- 世界书命中条目展示。
- 记忆注入与写入原因展示。
- 好感度变化日志。
- Provider 调用状态展示。

验收标准：

- 任意一次沙盒对话都能回放。
- 后台能展示本次回复使用了哪些上下文。
- 后台能说明本次是否保存记忆以及原因。
- 运营者无需读代码即可判断回复异常原因。

### Milestone 3：真实论坛灰度上线

周期建议：2–3 周。

目标：在真实 Flarum 环境中小范围、安全上线。

交付物：

- 白名单用户/板块配置。
- 人工审核开关。
- 敏感内容升级提醒。
- 发帖失败重试与失败记录。
- 回复频率策略。
- 线上日志审计页面。

验收标准：

- 可限制 Bot 只在指定板块或指定用户范围内回复。
- 可开启“生成但不自动发帖”的审核模式。
- 自循环、刷屏、低价值内容被有效拦截。
- 真实环境连续运行一周无严重异常。

### Milestone 4：人格与质量体系

周期建议：2–4 周。

目标：把“树洞喵回复好不好”从主观感觉变成可测试、可调优的质量体系。

交付物：

- 回复质量评估集。
- 情绪支持、校园咨询、闲聊、危机表达、恶意刷屏等场景用例。
- Prompt/世界书版本记录。
- 人工评分表。
- 回复质量趋势报告。

验收标准：

- 每次 Prompt 或世界书调整前后可以对比效果。
- 危机表达场景回复符合安全边界。
- 闲聊和低价值内容不会污染长期记忆。
- 角色表达稳定，不频繁跑偏。

### Milestone 5：微信只读信息助手 MVP

周期建议：3–5 周。

目标：完成微信群只读采集、归档、分类与日报/周报，不做自动回复。

交付物：

- 微信文字消息 collector。
- JSONL 去重存储。
- 附件目录监听。
- PDF/docx/xlsx 解析。
- OCR 安全模式原型。
- 日报/周报生成器。

验收标准：

- 能稳定识别目标群并采集可见文本。
- 不发送消息、不点击未知按钮、不修改微信设置。
- 可生成结构化日报/周报。
- 附件能归档并提取主要文本。
- 所有采集动作有本地日志记录。

### Milestone 6：统一校园 AI 助手平台

周期建议：中长期。

目标：将论坛回复、微信摘要、知识库、记忆画像和后台管理统一。

交付物：

- 统一用户画像模型。
- 多渠道事件总线。
- 统一知识库与世界书管理。
- 多 Agent 任务编排。
- 权限、审计、数据导出与删除机制。

验收标准：

- 同一套 AI 能力可服务 Flarum、微信摘要和后续渠道。
- 各渠道数据边界清晰，不混淆隐私与上下文。
- 管理后台可统一配置、测试、发布和审计。

## 6. 验收标准总表

| 方向 | 验收标准 | 推荐验证方式 |
|------|----------|--------------|
| 稳定性 | 核心测试可自动运行且通过 | `pytest`、`python sandbox_test.py` |
| 安全性 | 不自循环、不刷屏、不污染正式数据 | 沙盒测试、Webhook 回放、正式环境灰度 |
| 可解释性 | 可查看 Prompt、世界书、记忆、Provider、好感度 | 后台链路追踪页 |
| 可运营性 | 非开发者能调整世界书与测试用例 | Streamlit 后台操作验收 |
| 回复质量 | 关键场景回复符合角色与安全边界 | 质量评估集 + 人工评分 |
| 多渠道扩展 | 微信只读采集和日报周报可用 | 本地探测、JSONL 数据、报告生成 |
| 数据治理 | 支持用户数据查看、导出、删除和损坏备份 | 管理后台 + 文件检查 |

## 7. 技术任务拆解

### 7.1 测试体系

- 引入 `pytest` 作为统一测试入口。
- 将 `sandbox_test.py` 中的核心断言沉淀为可复用测试工具。
- 为 `core/worldbook.py` 增加关键词、正则、constant、禁用条目测试。
- 为 `core/providers/manager.py` 增加主备切换、失败恢复测试。
- 为 `core/memory/layered_memory.py` 增加并发写入、损坏文件备份、摘要触发测试。
- 为 `core/affection/affection_manager.py` 和 `core/affection/rewards.py` 增加奖励规则测试。
- 为 `api/webhook_listener.py` 增加典型 Flarum payload 回放测试。

### 7.2 回复链路追踪

- 定义 `GenerationTrace` 数据结构。
- 在 `core/llm_engine_v2.py` 中记录本次生成使用的世界书、记忆、Provider、Prompt 片段和保存结果。
- 在沙盒报告中写入 trace。
- 在 `admin_ui.py` 中增加 trace 查看和回放组件。
- 对真实环境 trace 做脱敏处理。

### 7.3 后台增强

- 增加沙盒测试用例编辑器。
- 增加 Prompt 预览功能。
- 增加世界书命中解释。
- 增加用户记忆删除确认与导出。
- 增加好感度变化历史。
- 增加 Provider 健康状态与调用统计。

### 7.4 灰度上线保护

- 增加板块白名单和用户白名单。
- 增加“只生成不发帖”的人工审核模式。
- 增加敏感内容识别与管理员提醒。
- 增加发帖失败重试与失败队列。
- 增加回复频率分级策略。
- 增加线上运行日志审计。

### 7.5 记忆与数据治理

- 明确 V2 JSON 记忆与 ChromaDB 记忆的边界。
- 增加记忆写入评分解释。
- 增加用户数据导出。
- 增加用户数据彻底删除。
- 增加摘要版本记录。
- 增加数据目录巡检脚本。

### 7.6 微信助手 MVP

- 完成 `wechat_group_assistant/collector` 模块。
- 设计消息 JSONL schema。
- 实现低频采集和去重。
- 实现附件目录监听。
- 实现 PDF/docx/xlsx 文档解析。
- 实现日报/周报 summarizer。
- 保持只读安全边界，不实现自动回复。

## 8. 近期两周执行计划

### 第 1–3 天：测试基建

- 创建 pytest 目录结构。

- 定义回复 trace 数据结构。

### 第 7–9 天：灰度保护

- 增加白名单/板块范围控制。
- 增加人工审核模式。
- 增加敏感内容升级策略。

### 第 10–14 天：微信只读采集起步

- 基于现有探测脚本实现 collector 原型。
- 输出去重 JSONL。

## 9. 风险与约束

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| Prompt 跑偏 | 回复不稳定或不符合人设 | 建立质量评估集和版本记录 |
| 真实论坛误发 | 影响用户体验和信任 | 沙盒先行、灰度、人工审核 |
| 记忆污染 | 长期上下文质量下降 | 记忆价值过滤、写入解释、人工删除 |
| 自循环或刷屏 | API 成本和论坛污染 | Bot 身份识别、冷却、频率策略 |
| 敏感内容处理不当 | 安全与伦理风险 | 危机场景模板、升级提醒、审计 |
| 微信自动化不稳定 | 采集失败或误操作 | 只读边界、低频采集、人工确认 |
| 多模块耦合 | 后续维护困难 | 明确接口、测试覆盖、模块化拆分 |

---

## 10. 产品北极星指标

建议长期关注以下指标：

- **有效回复率**：真实用户互动中被认为有帮助、无冒犯、无跑偏的回复占比。
- **安全拦截率**：自循环、刷屏、低价值、敏感内容被正确处理的比例。
- **记忆有效率**：写入长期记忆后，在后续对话中产生正向帮助的比例。
- **运营调试效率**：从发现异常回复到定位原因所需时间。
- **沙盒覆盖率**：关键场景是否都有沙盒或 pytest 用例覆盖。
- **多渠道信息利用率**：微信日报/周报中被用户实际阅读或采纳的信息比例。

## 11. 总结

Flarum AI Agent 当前已经具备“论坛 AI 陪伴角色”的核心技术骨架。下一阶段不应急于扩张功能，而应优先把它打磨成一个稳定、可解释、可测试、可运营的最小产品系统。

推荐路线是：

1. 先稳定核心链路和测试体系。
2. 再增强后台可解释性和运营能力。
3. 然后小范围灰度接入真实论坛。
4. 同步推进微信只读信息助手原型。
5. 最终沉淀为多渠道校园 AI 助手平台。

只要坚持“先沙盒，后真实；先解释，后自动化；先安全，再扩展”，这个项目就能从一个可用 Bot 逐步发展成一个真正可持续运营的校园 AI Agent 产品。
